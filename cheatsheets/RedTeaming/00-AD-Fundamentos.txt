===========================================================
  T2 - 0. FUNDAMENTOS DE ACTIVE DIRECTORY (AD)
===========================================================

===========================================================
  TERMINOLOGIA CLAVE
===========================================================
DC          Domain Controller     : servidor que gestiona el dominio
KDC         Key Distrib. Center   : servicio Kerberos (normalmente el DC)
NTDS.dit    Base de datos del DC  : contiene TODOS los hashes del dominio
SAM         Security Acct Manager : base de datos local de hashes (no-DC)
LSASS       Local Security Auth.  : proceso Windows que gestiona auth en memoria
SPN         Service Principal Name: identificador unico de un servicio
              ej: CIFS/servidor.lab.local
SID         Security Identifier   : ID unico de usuario/grupo en el dominio
              ej: S-1-5-21-<dominio>-<RID>
TGT         Ticket Granting Ticket: "pasaporte" para pedir tickets de servicio
TGS / ST    Service Ticket        : ticket de acceso a un servicio concreto
krbtgt      Cuenta especial del DC: su hash cifra todos los TGT del dominio

===========================================================
  ACTORES EN CADA AUTENTICACION
===========================================================
Cliente     : maquina/usuario que quiere acceder
DC / KDC    : autoridad central que autentica y emite tickets
Servidor    : recurso/servicio al que se quiere acceder (CIFS, HTTP, MSSQL...)

===========================================================
  FLUJO NTLM (Challenge-Response)
===========================================================

  Cliente                Servidor               DC
    |                       |                    |
    |-- [1] NEGOTIATE ----→ |                    |
    |   (usuario)           |                    |
    |                       |                    |
    |← [2] CHALLENGE -------|                    |
    |   (nonce aleatorio)   |                    |
    |                       |                    |
    |-- [3] RESPONSE -----→ |                    |
    |   NTLM_hash(nonce)    |                    |
    |                       |                    |
    |                       |-- [4] VERIFY ----→ |
    |                       |   user+nonce+resp  |
    |                       |                    |
    |                       |← [5] OK / KO ------|
    |                       |   (compara hashes) |

--- ATAQUES POR PASO ---
[1-2]  Responder / NTLMrelayx : envenenamiento LLMNR/NBT-NS
       -> el atacante responde en lugar del servidor legítimo
       -> el cliente manda la RESPONSE al atacante (paso 3)
       -> esa RESPONSE es el NetNTLMv2 crackeable (hashcat -m 5600)

[3-4]  NTLM Relay (NTLMrelayx): reenvía la RESPONSE capturada
       -> no hace falta crackear, se retransmite directamente a otro host

[3]    Pass-the-Hash (PtH)    : inyecta el hash NTLM directamente
       -> salta los pasos 1-3, el hash ES la credencial
       -> herramienta: mimikatz / impacket-psexec / xfreerdp

[5]    Internal Monologue     : fuerza NTLM local sin red
       -> extrae NetNTLMv2 sin necesidad de admin ni Mimikatz

===========================================================
  FLUJO KERBEROS (Tickets)
===========================================================

  Cliente                KDC                 Servidor
    |                     |                     |
    |-- [1] AS-REQ -----→ |                     |
    |   user + timestamp  |                     |
    |   (cifrado c/ hash) |                     |
    |                     |                     |
    |← [2] AS-REP --------|                     |
    |   TGT               |                     |
    |   (cifrado krbtgt)  |                     |
    |                     |                     |
    |-- [3] TGS-REQ ----→ |                     |
    |   TGT + SPN         |                     |
    |                     |                     |
    |← [4] TGS-REP -------|                     |
    |   ST                |                     |
    |   (cifrado hash SPN)|                     |
    |                     |                     |
    |-- [5] AP-REQ -------------------------→   |
    |   Service Ticket    |                     |
    |                     |                     |
    |← [6] ACCESO CONCEDIDO ------------------- |
    |   (sin contactar DC)|                     |

--- ATAQUES POR PASO ---
[1]   ASREPRoasting        : si el usuario NO tiene pre-autenticacion
      -> se pide AS-REP sin credenciales, parte del AS-REP es crackeable
      -> hashcat -m 18200
      -> herramienta: impacket-GetNPUsers / Rubeus asreproast

[2]   Golden Ticket        : si se obtiene el hash de krbtgt
      -> se falsifica un TGT para cualquier usuario del dominio
      -> persiste aunque se cambien contraseñas (hasta rotar krbtgt x2)
      -> herramienta: mimikatz kerberos::golden

[3]   Kerberoasting        : solo requiere un usuario de dominio valido
      -> cualquiera puede pedir TGS de cuentas con SPN
      -> el TGS viaja cifrado con el hash de la cuenta de servicio
      -> se crackea offline: hashcat -m 13100 (RC4) / -m 19700 (AES)
      -> herramienta: impacket-GetUserSPNs / Rubeus kerberoast

[4]   Silver Ticket        : si se obtiene el hash de la cuenta de maquina
      -> se falsifica un ST para esa maquina+servicio concreto
      -> no requiere contactar al KDC (mas sigiloso que Golden)
      -> herramienta: mimikatz kerberos::golden (mismo modulo)

[5]   Pass-the-Ticket (PtT): se inyecta un ST o TGT robado
      -> se roba de memoria con Mimikatz o de /tmp en Linux (ccache)
      -> herramienta: mimikatz kerberos::ptt / export KRB5CCNAME

[6]   Unconstrained Deleg. : el ST del paso 5 incluye el TGT del usuario
      -> maquinas con esta config cachean TGTs de quien se conecte
      -> comprometer esa maquina = robar TGTs de admins
      -> herramienta: Rubeus monitor + kerberos::ptt

===========================================================
  NOTAS RAPIDAS
===========================================================
- RC4 vs AES en Kerberos: $23 = RC4 (mas debil), $18 = AES
- NTLM hash = MD4(contraseña). No necesita sal -> rainbow tables
- NetNTLMv2 != NTLM hash. NetNTLMv2 no se puede usar en PtH
- LM hash: version antigua y muy debil. En Win moderno suele ser
  aad3b435b51404eeaad3b435b51404ee (hash vacio, ignorar)
- krbtgt debe rotarse 2 veces para invalidar Golden Tickets
- Silver Ticket no toca el KDC -> menos logs, mas sigiloso
